# A11y-01: WCAG Accessibility Compliance Test

TEST ID: A11y-01 (Accessibility)
PURPOSE: Validate basic WCAG 2.1 Level AA compliance
EXPECTED: Meets minimum WCAG targets; navigable without mouse

=== WCAG 2.1 LEVEL AA REQUIREMENTS ===

1. Perceivable
--------------
Information must be presentable to users in ways they can perceive.

1.1 Text Alternatives:
□ All images have alt text
□ Icons have aria-labels
□ Decorative images: alt="" or role="presentation"

1.2 Time-based Media:
□ Videos have captions (if applicable)
□ Audio descriptions provided (if applicable)

1.3 Adaptable:
□ Content structure uses semantic HTML
□ Headings in logical order (h1 > h2 > h3)
□ Lists use <ul>, <ol>, <li>
□ Forms use <label> for inputs
□ Tables use <th>, <caption> properly

1.4 Distinguishable:
□ Color contrast ratio ≥ 4.5:1 (text)
□ Color contrast ratio ≥ 3:1 (UI components)
□ Text resizable up to 200% without loss
□ No information conveyed by color alone
□ Background audio controllable

2. Operable
-----------
UI components must be operable.

2.1 Keyboard Accessible:
□ All functionality via keyboard
□ No keyboard traps
□ Visible focus indicator
□ Skip to main content link

2.2 Enough Time:
□ Time limits adjustable/extendable
□ Auto-refresh can be paused
□ Timeouts give warnings

2.3 Seizures:
□ No content flashes more than 3 times/second

2.4 Navigable:
□ Page titles descriptive
□ Focus order logical
□ Link purpose clear from text
□ Multiple ways to find pages
□ Headings and labels descriptive
□ Keyboard focus visible

2.5 Input Modalities:
□ Touch targets ≥ 44x44 pixels
□ Motion-based actions have alternatives

3. Understandable
-----------------
Information must be understandable.

3.1 Readable:
□ Language of page identified (lang="en"/"he")
□ Language changes marked
□ Terms defined or explained

3.2 Predictable:
□ Focus doesn't trigger unexpected changes
□ Input doesn't cause unexpected changes
□ Navigation consistent across pages
□ Components identified consistently

3.3 Input Assistance:
□ Error messages clear and helpful
□ Labels or instructions provided
□ Error suggestions provided
□ Error prevention (confirm before delete)

4. Robust
---------
Content must be robust enough for assistive technologies.

4.1 Compatible:
□ Valid HTML (no parsing errors)
□ Name, role, value for all UI components
□ Status messages programmatically determinable

=== KEYBOARD NAVIGATION TEST ===

Tab Navigation Order:
---------------------
1. Skip to main content link
2. Main navigation menu
3. Search field
4. Primary content area
5. Form fields (logical order)
6. Action buttons
7. Footer links

Verify:
□ Tab order matches visual layout
□ No hidden interactive elements in tab order
□ Focus visible on all elements
□ Shift+Tab reverses order

Focus Indicators:
-----------------
All interactive elements must show focus:
□ Buttons: visible outline/border
□ Links: underline or outline
□ Form fields: border highlight
□ Checkboxes/radios: outline
□ Dropdowns: border change

Minimum: 2px solid border, high contrast color

=== SCREEN READER TESTING ===

NVDA (Windows):
---------------
Test with NVDA (free):
1. Launch NVDA
2. Navigate site with keyboard
3. Verify announcements:
   □ Page title announced
   □ Headings announced with level
   □ Form labels read correctly
   □ Button states announced
   □ Error messages announced
   □ Dynamic content updates announced (aria-live)

Common Issues:
--------------
✗ Missing alt text: "Image" or blank
✗ Unlabeled buttons: "Button" only
✗ Form without labels: field purpose unclear
✗ Table without headers: data unstructured

Correct Announcements:
---------------------

✓ "Session history, heading level 2"

=== SEMANTIC HTML TEST ===

Correct Structure:
------------------
<header>
  <nav aria-label="Main navigation">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/files">Files</a></li>
    </ul>
  </nav>
</header>

<main>
  <h1>Session History</h1>
  <section aria-labelledby="recent">
    <h2 id="recent">Recent Sessions</h2>
    <ul>...</ul>
  </section>
</main>

<footer>
  <p>&copy; 2025 AI Adama</p>
</footer>

Avoid:
------
✗ <div> for everything
✗ Click handlers on non-interactive elements
✗ Images without alt
✗ Forms without labels
✗ Fake buttons (<div onclick="...">)

=== ARIA USAGE ===

When to Use:
------------
□ aria-label: When visible label insufficient
□ aria-describedby: Additional descriptions
□ aria-live: Dynamic content updates
□ aria-expanded: Collapsible sections
□ aria-current: Current page in navigation
□ role: When semantic HTML unavailable

Examples:
---------
<button aria-label="Close dialog">✕</button>
<input aria-describedby="pwd-help" />
<div role="alert" aria-live="assertive">Error occurred</div>
<button aria-expanded="false" aria-controls="menu">Menu</button>
